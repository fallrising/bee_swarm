# Bee Swarm Project Context

## 🎯 專案核心定位

**Bee Swarm** 是一個基於 GitHub 的 **AI 團隊協作概念設計與模擬框架**，專注於探索 AI 角色如何透過異步分工實現高效協作。

### 關鍵特性
- **概念設計工具**：專注於協作模式設計，而非生產系統實現
- **模擬驗證平台**：通過 SimPy 模擬器驗證協作效果和性能
- **GitHub-Centric 架構**：完全基於 GitHub 現有功能實現協調
- **異步協作範式**：AI 角色輪流處理任務，避免複雜的即時通信

## 🧠 核心設計哲學

### 1. 簡化優先 (Simplicity First)
- 移除不必要的複雜性，專注核心功能
- 利用現有成熟工具，避免重複造輪子
- 最小化維護成本

### 2. 透明性 (Transparency)
- 所有過程在 GitHub 上可見
- 完整的版本控制歷史
- 清晰的審計軌跡

### 3. 異步協作 (Asynchronous Collaboration)
- 避免即時通信的複雜性
- 輪流處理模式
- 狀態驅動的工作流

### 4. 可驗證性 (Verifiability)
- 模擬先行，通過 SimPy 驗證設計效果
- 數據驅動的設計決策
- 持續改進

## 📐 設計原則

### 1. 簡化優先原則
- 能用現有工具解決就不自己開發
- 能用簡單方案解決就不用複雜方案
- 能減少組件就不增加組件

### 2. 透明性原則
- 所有協調過程在 GitHub 上可見
- 決策過程有完整記錄
- 狀態變化有明確軌跡

### 3. 可驗證性原則
- SimPy 模擬器驗證協作效果
- A/B 測試比較不同方案
- 量化指標評估效果

### 4. 可擴展性原則
- 角色模塊化，易於添加新角色
- 工具接口標準化
- 不為不確定需求過度設計

## 🚀 核心價值

### ✅ 概念先行 (Concept First)
專注於協作模式的設計與驗證，而非技術實現細節

### ✅ 數據驅動 (Data-Driven)

## 🔄 混合架構思考

### 當前架構：長時間存活容器
- 各角色獨立容器，持續運行
- 資源持續占用，需要容器編排

### 新思路：GitHub Actions 自動化
- 事件驅動執行，按需分配資源
- 零基礎設施成本，完全基於 GitHub

### 混合模式潛力
- **核心協調**：保留容器架構處理複雜任務
- **輕量任務**：使用 GitHub Actions 處理簡單任務
- **成本優化**：根據任務複雜度選擇執行方式
- **漸進遷移**：逐步驗證和遷移適合的任務
